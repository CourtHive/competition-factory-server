# Server Architecture

The Competition Factory Server is built on NestJS 11, using a modular architecture with dependency injection, WebSocket gateway, and pluggable storage.

## Module Structure

```
AppModule
├── ConfigModule          (global) — environment variables, Joi validation
├── StorageModule          (global) — pluggable storage (LevelDB / PostgreSQL)
├── CacheModule            (global) — Redis caching via @nestjs/cache-manager
├── AuthModule             — JWT authentication, login, registration
├── UsersModule            — user CRUD, role management
├── ProvidersModule        — provider/organisation management
├── FactoryModule          — competition-factory engine integration
│   ├── FactoryController  — REST API endpoints
│   └── FactoryService     — business logic, engine management
├── MessagingModule        — WebSocket gateway
│   └── TmxGateway         — Socket.IO /tmx namespace
└── EmailModule            — Mailgun email service
```

## Request Flow (REST)

```
HTTP Request
  → Express middleware (body-parser, CORS)
    → NestJS route guards (JWT auth, role checks)
      → Controller method
        → Service (business logic)
          → Factory engine (tods-competition-factory)
          → Storage interface (read/write)
        → Response
```

## Request Flow (WebSocket)

```
Socket.IO message (/tmx namespace)
  → TmxGateway.messageHandler()
    → tmxMessages.executionQueue()
      → withTournamentLock()
        → fetchTournamentRecords()
        → mutationEngine.executionQueue(methods)
        → saveTournamentRecords()
      → client.emit('ack')
```

## Per-Request State Isolation

The server uses Node.js `async_hooks` (via `asyncGlobalState.ts`) to provide per-request state isolation for the factory engine. Each incoming request gets its own engine state context, preventing cross-request data leakage.

## Concurrency Control

`tournamentMutex.ts` provides per-tournament async locking:
- Sorted key acquisition prevents deadlocks
- 30-second timeout prevents stuck locks
- Multiple tournaments in a single request are locked in deterministic order
