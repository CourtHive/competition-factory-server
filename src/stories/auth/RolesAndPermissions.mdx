# Roles & Permissions

The server uses a role-based access control (RBAC) model. Users are assigned roles and fine-grained permissions at the user level, with provider-scoped authorization on top.

## Provider Scoping

Users are associated with a provider (organisation). When logged in:
- An `admin` can manage users and tournaments **within their provider only**
- A `superadmin` can manage **all providers and users**
- A `client` can access tournaments **published by their provider** or tournaments they own

## Managing Roles

```bash
# Set roles for a user
node src/scripts/admin-user.mjs set-roles \
  --email user@example.com \
  --roles admin,client,score

# Demote to client-only
node src/scripts/admin-user.mjs set-roles \
  --email user@example.com \
  --roles client
```

## JWT Token Structure

On login, the server issues a JWT containing:

```json
{
  "email": "user@example.com",
  "roles": ["admin", "client"],
  "permissions": { "deleteTournament": true },
  "services": { "tournamentProfile": true },
  "providerId": "uuid-of-provider"
}
```

The token is valid for `JWT_VALIDITY` (default: 2 hours).
