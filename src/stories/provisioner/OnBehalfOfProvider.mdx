# Provisioner — Acting on Behalf of a Provider

A provisioner manages many providers. To read or write a **specific** provider's
data, the provisioner authenticates with its own API key and names the target
provider with an `X-Provider-Id` header. The server then treats the request as a
**PROVIDER_ADMIN of that provider** for the duration of the call.

> **Provider key vs. provisioner key.** A provider key (`pkey_live_*`) is bound to
> a single provider — see **Provider API/Authentication**. A provisioner key
> (`prov_sk_live_*`) can act on behalf of **any** provider it owns or has a
> subsidiary relationship with, chosen per request via `X-Provider-Id`. The
> low-level auth mechanics live in **Provisioner/Authentication**; this page is the
> task-focused walkthrough.

## Prerequisites

- A provisioner API key (`prov_sk_live_*`), issued by a CourtHive super-admin
  (see **Provisioner/Admin API**).
- An **owner** or **subsidiary** relationship between your provisioner and the
  target provider. Without a valid relationship the `X-Provider-Id` header is
  ignored and provider-scoped calls are rejected.

## The two headers

```
Authorization: Bearer prov_sk_live_<your-key>
X-Provider-Id: <providerId>
```

Server-side, the middleware validates the key, looks up the provisioner↔provider
relationship, and — if valid — injects a synthetic `PROVIDER_ADMIN` user context
scoped to that provider. Tournament and scoring endpoints then behave as if a provider
admin of that provider had called them.

> **The synthetic role is scoped to tournament/scoring work, not account administration.**
> A provisioner **API key** cannot create or modify users, reset passwords, or edit
> `user_providers` associations through the console admin endpoints — those reject API-key
> callers with **403**. Provision end-user accounts via `POST /provisioner/users` (SSO). See
> **Provisioner/Authentication → Authorization boundary** for the full rule.

If you **omit** `X-Provider-Id`, you are authenticated as the provisioner but have
no provider scope — only the provisioner-native `/provisioner/*` management
endpoints work.

## Step 1 — Discover your providers

```bash
curl -X GET https://courthive.net/provisioner/providers \
  -H "Authorization: Bearer prov_sk_live_<your-key>"
```

Returns the providers your provisioner manages (each with its `organisationId`,
abbreviation, name, and relationship). Pick the `organisationId` you want to act
as — that is the value for `X-Provider-Id`.

## Step 2 — Create or update a tournament on behalf of a provider

Use the standard factory **save** endpoint with `X-Provider-Id` set. The
tournament's `parentOrganisation.organisationId` must match that provider.

```bash
curl -X POST https://courthive.net/factory/save \
  -H "Authorization: Bearer prov_sk_live_<your-key>" \
  -H "X-Provider-Id: 24bf9f25-96ca-401e-9660-f5571ebc50ba" \
  -H "Content-Type: application/json" \
  -d '{
        "tournamentRecord": {
          "tournamentId": "a1f2…",
          "tournamentName": "Battle of Boca - June",
          "startDate": "2026-06-01",
          "endDate": "2026-06-04",
          "parentOrganisation": { "organisationId": "24bf9f25-96ca-401e-9660-f5571ebc50ba" }
        }
      }'
```

This is the same `saveTournamentRecords` path TMX and provider keys use; the
provisioner simply supplies the provider scope via the header. Saving keeps the
provider's calendar in sync automatically.

## Step 3 — Read a tournament

```bash
curl -X POST https://courthive.net/factory/fetch \
  -H "Authorization: Bearer prov_sk_live_<your-key>" \
  -H "X-Provider-Id: 24bf9f25-96ca-401e-9660-f5571ebc50ba" \
  -H "Content-Type: application/json" \
  -d '{ "tournamentId": "a1f2…" }'
```

Reads are scoped to what your injected provider context permits.

## Step 4 — Run mutations (draws, scoring, scheduling…)

The canonical mutation path is the **execution queue** — `POST /factory` — which
applies an ordered list of factory methods. On tournament creation it also
**stamps ownership** to your provisioner + provider (the `tournament_provisioner`
record), so tournaments you provision are tracked as yours.

```bash
curl -X POST https://courthive.net/factory \
  -H "Authorization: Bearer prov_sk_live_<your-key>" \
  -H "X-Provider-Id: 24bf9f25-96ca-401e-9660-f5571ebc50ba" \
  -H "Content-Type: application/json" \
  -d '{
        "tournamentId": "a1f2…",
        "methods": [
          { "method": "addEvent", "params": { "event": { "eventName": "Men'\''s Singles" } } }
        ]
      }'
```

## Relationship scope: owner vs. subsidiary

<table>
  <thead>
    <tr><th style={{ textAlign: 'left' }}>Relationship</th><th style={{ textAlign: 'left' }}>What you can do on behalf of the provider</th></tr>
  </thead>
  <tbody>
    <tr><td><code>owner</code></td><td>Full control — provider config &amp; caps, subsidiary management, and all tournament data operations.</td></tr>
    <tr><td><code>subsidiary</code></td><td>Scoped to your own tournaments within the provider. Cannot change provider config, caps, or subsidiaries (those are <code>ProvisionerOwnerGuard</code>-protected).</td></tr>
  </tbody>
</table>

## Errors

<table>
  <thead>
    <tr><th style={{ textAlign: 'left' }}>Status</th><th style={{ textAlign: 'left' }}>Cause</th></tr>
  </thead>
  <tbody>
    <tr><td><code>401 Unauthorized</code></td><td>Missing, malformed, revoked, or expired provisioner key.</td></tr>
    <tr><td><code>401</code> / <code>403</code></td><td>A provider-scoped (<code>/factory/&#42;</code>) call with no <code>X-Provider-Id</code>, or an <code>X-Provider-Id</code> you have no relationship to — no provider context is injected, so authorization fails.</td></tr>
    <tr><td><code>403</code></td><td>An owner-only <code>/provisioner/&#42;</code> operation attempted with a <code>subsidiary</code> relationship.</td></tr>
    <tr><td><code>403</code></td><td>A user-administration endpoint (<code>/auth/admin-create-user</code>, <code>/auth/admin-reset-password</code>, <code>/auth/modify</code>, or a <code>user_providers</code> association mutation) called with an API key — the synthetic <code>PROVIDER_ADMIN</code> does not grant user administration. Use <code>POST /provisioner/users</code> instead.</td></tr>
  </tbody>
</table>
